Design: Abschnitt 3/4 — Tod, Revival & Item-Drops

Character-Lebenszyklus und das Item-System

Tod-Sequenz

Construct → sendet Counter-Attack-Message → Character
Character: CHAR_HP -= damageAmount
if CHAR_HP ≤ 0:
CHAR_IS_DEAD = 1
penalty: random 1–3 Skillpunkte von random Attributen abziehen
Character nimmt keine Angriffs-TXs mehr an (refund)

Penalty-Randomness via getWeakRandomNumber() — kein Attribut kann unter 0 fallen.

Revival

Owner → sendet Revival-Token an Character
Character: prüft assetId == REVIVAL_TOKEN_ID
CHAR_IS_DEAD = 0
CHAR_HP = CHAR_MAX_HP (volle HP)
Revival-Token wird konsumiert (gesendet an Burn-Adresse oder Creator)

Revival-Token ID ist beim Deploy konfigurierbar (ähnlich xpTokenId beim Construct).

Item-Drops

Construct prüft nach erfolgreichem Character-Angriff ob ein Drop ausgelöst wird:

drop_chance = BASE_CHANCE + (luck * LUCK_BONUS_PER_POINT)
if getWeakRandomNumber() % 100 < drop_chance:
// welches Item?
balance = getAssetBalance(this, dropTokenId)
if balance > 0:
sendQuantity(1, dropTokenId, characterSender)
Construct kann mehrere Drop-Pools haben (KKV-Map: dropSlot → tokenId), konfigurierbar durch Creator. getAssetBalance prüft ob noch Vorrat vorhanden.

Items verwenden / weiterhandeln

Im Kampf einsetzen

Owner schickt Item-Token mit Angriffs-TX an Character → Character leitet es weiter an Construct (wie heutiger Damage-Modifier-Flow)

Handeln

Owner ruft COLLECT_ITEMS auf Character auf → Character sendet alle (oder bestimmte) Item-Tokens an Owner-Wallet → DeFi-Trading

Migration (vereinfacht)

Owner → ruft MIGRATE(oldCharId) auf neuem Contract auf
Neuer Contract: liest Attribute aus altem KKV → kopiert in eigene KKV
Owner → Emergency Withdraw auf altem Contract → Tokens/SIGNA zurück
Owner → lädt neuen Contract mit Tokens auf
