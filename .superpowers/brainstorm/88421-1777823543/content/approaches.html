3 Architektur-Ansätze: Character Contract

Wie soll der Character Contract mit dem Construct interagieren?

A

Thin Character / Smart Construct ★ Empfehlung

Character ist einfacher Zustandsspeicher + Proxy. Construct bleibt das "Gehirn".

  • Character speichert: Attribute, HP, Statuseffekte (KKV), Token-Inventory
  • Attack-Message enthält Attribute-Werte → Construct berechnet alles
  • Construct sendet Counter-Attack/Buff/Drop-Messages zurück
  • Character updated seinen Zustand basierend auf eingehenden Messages

Vorteile

  • Construct bleibt autoritativ
  • Character-Contract einfach & klein
  • Bestehende Construct-Logik leicht erweiterbar

Nachteile

  • Construct muss Character-Angriffe erkennen
  • Attribute könnten gefälscht werden (braucht Validierung)
B

Fat Character / Dumb Construct

Character hat volle Kampflogik, Construct wird vereinfacht.

  • Character berechnet modifizierten Schaden selbst (mit Attributen)
  • Sendet berechneten Schaden + SIGNA an Construct
  • Construct vertraut dem Character-Ergebnis (nur Breach-Limit check)
  • Mehr Autonomie für Character

Vorteile

  • Klare Trennung: Character-Logik in Character
  • Construct bleibt einfach

Nachteile

  • Zwei Contracts für Schadensberechnung schwer zu synchronisieren
  • Sicherheitsrisiko: Character könnte überhöhten Schaden melden
C

Registry-validierter Character

Zentraler Registry-Contract validiert alle Characters. Construct prüft Registry vor Annahme.

  • Character Registry speichert: Typ A, deployer, creation-block
  • Construct prüft Registry: "Ist dieser Sender ein gültiger Character?"
  • Dann wie Ansatz A: Attribute in Message, Construct berechnet
  • Registry kann auch Level-Schwellenwerte & Fähigkeits-Konfiguration speichern

Vorteile

  • Sicherheit: kein Fake-Character möglich
  • Registry kann konfigurierbare Level-Tabelle, Fähigkeiten etc. speichern
  • Zukunftssicher für neue Character-Typen

Nachteile

  • Extra Contract = extra Deployment & Kosten
  • Jeden Angriff muss Construct Registry abfragen (geht nicht on-chain, braucht Trick)

Meine Empfehlung

A + Teile von C: Character als Thin Proxy (A), aber mit einer einfachen Validierung: Construct prüft ob der Sender-Account in einer KKV-Map als "registrierter Character" eingetragen ist — kein eigener Registry-Contract nötig, nur eine Admin-Funktion im Construct.
