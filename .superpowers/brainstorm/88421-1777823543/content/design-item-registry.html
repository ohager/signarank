Item Registry — Design

Globaler Contract für Character-Items (Equipment & Consumables)

Abgrenzung: was bleibt, was ist neu

Unverändert

Angriffs-Modifier-Tokens (Laser, Eis, Feuer)
→ bleiben in Construct-KKV
→ per-Construct konfigurierbar

Neu: Item Registry

Character-Items (Schattendolch, Heilpotion…)
→ globaler Registry-Contract
→ gelesen von Character + Construct

Item Registry KKV-Struktur (key1=property, key2=tokenId)

PROP_ITEM_TYPE 1 // 1=equipment, 2=consumable
PROP_EFFECT_TARGET 2 // 0=attack, 1=HP, 2=STR, 3=STA, 4=DEX, 5=LCK, 6=WIL
PROP_BONUS_ABS 3 // absoluter Bonus z.B. +50 HP, +1 Stärke
PROP_BONUS_REL 4 // relativer Bonus: 120=+20%, 90=–10% (Malus)
PROP_STACK_LIMIT 5 // max im Inventory: 1=unique, N=stackable
PROP_MIN_LEVEL 6 // Mindeststufe zum Tragen
PROP_IS_BURNABLE 7 // 1=wird nach Anwendung verbrannt (to Creator)

Inventory = Token-Balance

Kein separates KKV-Tracking nötig — getAssetBalance(this, tokenId) IS das Inventory.

// Character empfängt Item-Token:
long limit = getMapValue(REGISTRY_ID, PROP_STACK_LIMIT, tokenId);
long current = getAssetBalance(this, tokenId);
long incoming = getQuantity(currentTx.txId, tokenId);
if (current >= limit) → refund all
else if (current + incoming > limit) → refund excess
else → behalten (bleibt als Token im Contract)

Consumables (Burnable Items)

Owner → USE_ITEM(tokenId) message an Character
Character liest Registry:
ITEM_TYPE == CONSUMABLE?
EFFECT_TARGET + BONUS_ABS → wende Effekt an
IS_BURNABLE == 1 → sendQuantity(1, tokenId, Creator)
Beispiel Heilpotion: MAP_CHAR_HP += BONUS_ABS (max MAX_HP)

Equipment-Items: Wie werden sie applied?

A

Passive via Token-Balance

Construct liest bei jedem Angriff: "Hat Character dieses Equipment-Token?" via getAssetBalance(sender, tokenId). Liest dann Effekt aus Registry.

Problem: Construct muss wissen welche tokenIds als Equipment existieren → braucht eine Liste

B

Aktiv: Character schreibt aggregierten Bonus in KKV

Wenn Character ein Equipment empfängt, liest er Registry und schreibt den aggregierten Effekt in seine KKV (z.B. MAP_CHAR_EQUIP_ATK_BONUS).

Construct liest nur 1-2 KKV-Werte statt vieler Token-Balances. Einfacher für den Construct.

Meine Empfehlung: B — Character aggregiert Equipment-Boni in KKV, Construct liest nur die aggregierten Werte.
